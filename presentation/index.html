La Plateforme - Student Tracker

Présentation du Projet

Objectifs du projet

  • Développer une application Java complète avec un backend (CRUD), un frontend graphique, et des tests automatisés.
  • Assurer la sécurité des données et des accès grâce à l’authentification JWT.
  • Permettre le déploiement professionnel de l’application sur le cloud.

Fonctionnalités du projet

  • 🔐Authentification des utilisateurs (JWT)
  • 📝Gestion complète des étudiants (ajout, modification, suppression, consultation)
  • 📊Tableau de bord analytique
  • 📄Exportation des données en PDF
  • 🔎Recherche et tri dynamiques
  • 💻Interface graphique moderne (JavaFX)

Architecture modulaire du projet

Le projet Student Tracker adopte une architecture modulaire. Le backend Java est organisé en couches distinctes (DAO, Service, Handler, Model, Util) pour garantir la séparation des responsabilités, la maintenabilité et la testabilité. Le frontend JavaFX offre une interface graphique, connectée au backend via une API REST sécurisée par JWT. Cette structure permet un développement évolutif, des tests automatisés, et un déploiement cloud efficace.

Fonctionnement du backend

Le backend Java est structuré en couches : Handler (gestion des requêtes HTTP), Service (logique métier), DAO (accès aux données), Model (entités), Util (outils). Lorsqu’une requête HTTP arrive (ex : POST /students), elle est traitée par un Handler, qui délègue au Service, qui utilise le DAO pour accéder à la base PostgreSQL. Les réponses sont renvoyées en JSON.

Sécurité : Authentification JWT sur les endpoints protégés.

Exemples d’API :
POST /login — Authentification
GET /students — Liste des étudiants
POST /students — Ajouter un étudiant
POST /register — Inscription utilisateur

Technologies : Java 17, Maven, PostgreSQL, JWT, HTTP, JSON, JUnit

Fonctionnement du frontend

Le frontend est développé en JavaFX avec des vues FXML et suit le modèle MVC (Modèle-Vue-Contrôleur). Les contrôleurs gèrent les interactions utilisateur, la logique de navigation, la recherche, le tri, et l’export PDF. L’interface graphique est réactive et intuitive.

Communication : Le frontend communique avec le backend via l’API REST (HTTP/JSON) et gère l’authentification avec des tokens JWT.

Fonctionnalités principales : Authentification, gestion des étudiants, recherche et tri dynamiques, export PDF, tableau de bord analytique.

Technologies : JavaFX, FXML, CSS, HTTP, JSON, JWT

Intégration Backend & Frontend

Cliquez ici pour voir le schéma d’intégration

Fonctionnement de l’intégration

Le frontend JavaFX communique avec le backend via une API REST (HTTP/JSON). Toutes les opérations (authentification, gestion des étudiants, export PDF, analytics) passent par des requêtes HTTP.

Authentification : Après connexion, le backend fournit un token JWT. Ce token est utilisé par le frontend pour toutes les requêtes protégées.

Échanges de données : Les données (étudiants, résultats, etc.) sont échangées en JSON. L’export PDF se fait côté frontend à partir des données reçues.

Gestion des erreurs : Le backend renvoie des statuts HTTP et messages d’erreur, affichés à l’utilisateur par le frontend.

Tests & Couverture

Couverture actuelle

  • DAO : Tests JUnit sur StudentDAO et UserDAO (CRUD étudiants et utilisateurs)
  • Service : Tests JUnit/Mockito sur UserService (logique métier, authentification)
  • Util : Tests JUnit sur DatabaseManager (connexion à la base)
Outils : JUnit, Mockito, curl (pour tester l’API REST)

Améliorations à apporter

  • Ajouter des tests d’intégration bout-en-bout
  • Ajouter des tests pour le frontend (JavaFX)

Exemple de test JUnit

@Test
public void testAddStudent() {
    Student s = new Student("Alice", "Dupont", 21, 15.5);
    studentDAO.addStudent(s);
    assertNotNull(studentDAO.getStudentByID(s.getId()));
}

Déploiement

Étapes de déploiement

  • Préparer le code backend pour le cloud (Render.com)
  • Configurer la base PostgreSQL sur Render
  • Définir les variables d’environnement (JDBC, JWT)
  • Configurer les commandes de build et de démarrage (Maven, JAR)
  • Lancer le déploiement et tester l’API en ligne (curl)

Bonnes pratiques pour le déploiement

  • Utiliser des variables d’environnement pour la sécurité
  • Garder la version locale du backend
  • Tester chaque endpoint après déploiement
  • Surveiller les logs et l’état du service sur Render
